# 포인트 관리 시스템 PRD (Product Requirements Document)

## 1. 제품 개요

### 1.1 제품명
**포인트 관리 시스템**

### 1.2 제품 목적
사용자의 포인트 적립, 사용, 취소 등의 포인트 거래를 관리하고 계좌 정보를 제공하는 백엔드 시스템

### 1.3 제품 비전
- 안전하고 신뢰할 수 있는 포인트 거래 처리
- 유연하고 확장 가능한 포인트 지급 정책 관리
- 헥사고날 아키텍처 기반의 유지보수성 높은 시스템

## 2. 핵심 기능

### 2.1 포인트 거래 기능
- **포인트 적립**: 다양한 포인트 유형으로 포인트 적립
- **포인트 적립 취소**: 이미 적립된 포인트의 취소 처리
- **포인트 사용**: 주문에 대한 포인트 사용 처리
- **포인트 사용 취소**: 주문 취소에 따른 포인트 복원

## 3. 비즈니스 규칙

### 3.1 포인트 적립 규칙
- 1회 최소/최대 적립 금액 제한
- 개인 별 최대 보유 금액 제한
- 관리자 수기 지급이 가능하며 수기 지급은 다른 적립과 식별이 가능해야 한다.
- 적립 시 자동으로 만료일 계산 (현재일 + 유효기간)

### 3.2 포인트 사용 규칙
- 포인트 사용 순서 정책 : 설정 된 우선 순위 (관리자 수기 지급 > 일반 지급) > 잔여 만료일이 작은 순 > 먼저 적립 된 포인트
- 사용 가능한 포인트가 부족할 경우 거래 실패

### 3.4 포인트 취소 규칙
- **적립 취소**: 사용되지 않은 포인트만 취소 가능
- **사용 취소**: 주문별로 사용된 포인트 추적하여 취소
- **만료 포인트 처리**: 만료된 포인트 사용 취소 시 대체 지급


## 4. 설계 개념 및 프로세스 특징

### 4.1 핵심 설계 개념
- **도메인 중심 설계 (DDD)**: 비즈니스 로직을 `Domain Entity`와 `Processor`에 집중시켜 비즈니스 규칙의 응집도를 높이고 변경에 유연하게 대응합니다.
- **헥사고날 아키텍처**: 도메인을 외부 인프라(DB, API)로부터 격리하여 비즈니스 로직의 순수성을 유지하고 테스트 용이성을 확보합니다.
- **포인트 거래 및 상세 추적**: `PointTransaction`과 `PointTransactionDetail`을 분리하여 하나의 거래가 여러 입금분(`PointDeposit`)에 걸쳐 처리되는 과정을 정밀하게 기록합니다.
- **분산 식별자 전략 (TSID)**: 정렬 가능한 분산 ID인 `TSID`를 사용하여 대규모 환경에서도 고성능과 유연한 정렬 기능을 제공합니다.
- **데이터 정합성 및 동시성 제어**: 로컬 인메모리 락, DB 비관적/낙관적 락, 그리고 멱등성 키를 이용한 다중 보호 전략을 적용합니다. (상세 내용은 [10. 데이터 정합성 및 동시성 제어 전략](#10-데이터-정합성-및-동시성-제어-전략) 참고)

### 4.2 주요 프로세스 특징
- **선입선출(FIFO) 기반 차감**: 만료일이 임박한 포인트부터 우선적으로 차감하며, 동일 조건일 경우 먼저 적립된 포인트부터 소진합니다.
- **트랜잭션 이벤트 기반 구조**: 모든 포인트 처리는 이벤트를 통해 단계별로 기록되어, 복합 거래 및 취소 시의 추적성을 극대화합니다.
- **유연한 만료 처리**: 적립 시 정책에 따른 만료일을 자동 계산하며, 사용 취소 시 만료된 포인트에 대한 보정 로직을 포함합니다.
- **읽기/쓰기 데이터 소스 분리**: `RoutingDataSource`를 통해 DB 트래픽을 분산하고 확장성을 고려한 설정을 적용하였습니다.

## 5. 기술 아키텍처

### 5.1 아키텍처 패턴
**헥사고날 아키텍처 (Hexagonal Architecture)**
- **Domain Layer**: 비즈니스 로직과 규칙을 포함하는 핵심 계층
- **Application Layer**: 유스케이스 구현 및 도메인과 외부 시스템 연결
- **Infrastructure Layer**: 데이터베이스, 웹 API 등 외부 시스템과의 통신

## 9. 향후 확장 방향 (Future Expansion)

현재 프로젝트는 핵심 비즈니스 로직의 정합성과 확장성을 고려하여 설계되었으며, 향후 다음과 같은 방향으로 확장이 가능합니다.

### 9.1 고가용성 분산 락 도입
현재 사용자 레벨의 동시성 제어는 `ConcurrentHashMap`을 이용한 로컬 인메모리 락으로 구현되어 있습니다. 이는 단일 인스턴스 환경에서는 효율적이지만, 다중 인스턴스(Scale-out) 환경에서는 정합성을 보장하기 어렵습니다.
- **확장 방향**: Redis의 Redisson 라이브러리를 활용하여 분산 환경에서도 안정적인 **분산 락(Distributed Lock)** 체계로 전환할 수 있습니다.

### 9.2 비동기 이벤트 기반 및 이벤트 소싱 확장
현재 유스케이스와 도메인 모델은 대규모 트래픽 및 성능 확장에 대비하여 설계되었습니다.
- **확장 방향**: 성능 병목 시 동기 방식의 단일 트랜잭션에서 **이벤트 기반의 비동기 방식**으로 전환할 수 있습니다.
- **이벤트 소싱**: 현재 `PointTransaction`과 `PointTransactionEvent` 모델은 거래의 모든 상태 변화를 기록하고 있으므로, 이를 기반으로 도메인 상태를 재생성하는 **이벤트 소싱(Event Sourcing)** 아키텍처로 자연스럽게 확장 가능합니다.

### 9.3 트랜잭셔널 아웃박스(Transactional Outbox) 및 상태 머신
거래의 안정성과 타 서비스와의 연동 신뢰성을 높이기 위한 구조적 확장이 가능합니다.
- **아웃박스 패턴**: `PointTransactionEvent`를 아웃박스 테이블로 활용하여, 외부 메시지 브로커(Kafka 등)로의 이벤트 발행과 DB 트랜잭션의 원자성을 보장할 수 있습니다.
- **상태 머신 확장**: `PointTransaction` 모델을 정교한 상태 머신으로 관리하여, 복잡한 취소/재시도 워크플로우를 아웃박스 패턴과 결합해 안정적으로 제어할 수 있습니다.

### 9.4 CQRS를 통한 조회 성능 최적화
현재 `RoutingDataSource`를 통한 읽기/쓰기 분리를 넘어, 조회 전용 모델(Read Model)을 별도로 구축하여 복잡한 통계 및 대량 조회 성능을 극대화할 수 있습니다.

## 10. 데이터 정합성 및 동시성 제어 전략

본 시스템은 금융 거래와 유사한 포인트 데이터를 다루므로, 데이터의 무결성과 정합성을 보장하기 위해 다중 방어 전략을 사용합니다.

### 10.1 동시성 제어 전략 (Concurrency Control)

사용자의 포인트 잔액 및 입금 내역에 대한 동시 접근을 제어하기 위해 3단계 락 전략을 사용합니다.

1.  **애플리케이션 레벨: User Level Lock (UserLock)**
    - `@UserLock` 어노테이션과 `AOP`를 통해 사용자(`userId`)별로 동시 요청을 제어합니다.
    - `ConcurrentHashMap` 기반의 인메모리 락을 사용하여 동일 사용자가 짧은 시간에 여러 번의 거래를 시도할 때 발생할 수 있는 레이스 컨디션을 1차적으로 방어합니다.

2.  **데이터베이스 레벨: 비관적 락 (Pessimistic Lock)**
    - 포인트 차감 시(`PointUseService`), 차감 대상이 되는 `PointDeposit` 레코드를 `SELECT ... FOR UPDATE` 구문을 통해 잠금 처리합니다.
    - 트랜잭션이 종료될 때까지 다른 트랜잭션의 수정을 차단하여, 잔액 계산의 정확성을 보장합니다.

3.  **데이터베이스 레벨: 낙관적 락 (Optimistic Lock)**
    - 주요 엔티티(`PointAccount`, `PointDeposit`, `PointTransaction`)에 JPA `@Version`을 적용합니다.
    - 트랜잭션 커밋 시점에 데이터 변경 여부를 최종 확인하여, 만에 하나 발생할 수 있는 정합성 오류를 최종적으로 차단합니다.

### 10.2 멱등성 제어 전략 (Idempotency Control)

네트워크 재시도나 클라이언트의 중복 요청으로 인해 동일한 거래가 중복 처리되는 것을 방지합니다.

1.  **Idempotency Key 활용**
    - 모든 포인트 거래 요청 시 클라이언트로부터 고유한 `idempotencyKey`를 전달받습니다.
    - 이 키는 `PointTransactionEvent` 테이블에 저장되어 거래의 고유 식별자로 활용됩니다.

2.  **애플리케이션 기반 중복 검사**
    - 거래 처리 로직 진입 시, 해당 `userId`와 `idempotencyKey`로 이미 완료된 거래가 있는지 조회합니다.
    - 이미 처리된 거래인 경우, 새로 거래를 생성하지 않고 기존의 거래 결과(`Result.alreadyProcessed`)를 반환합니다.

3.  **데이터베이스 유니크 제약 조건 (Final Defense)**
    - `point_transaction_event` 테이블의 `idempotency_key` 컬럼에 `UNIQUE` 제약 조건을 설정합니다.
    - 애플리케이션 레벨의 체크를 통과한 동시 요청이 있더라도, DB 레벨에서 삽입 시점에 중복을 원천 차단하여 데이터 일관성을 유지합니다.
